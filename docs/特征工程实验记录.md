# 第一阶段特征工程实验记录

> 目标指标：**AUC**（排序能力，直接决定 MILP 能否为每条 query 选对模型）。
> ECE 作为**监控**指标（校准那步一般能压低），不作为优化目标。
> 口径：全量 1000 条，expanding 滚动回测，指标为**校准后** overall（`p_success_cal`），fold 内只用 `train_proper` 训练。

## 结论速览

| 版本 | 特征 | overall AUC | 单模型均值 AUC | ECE |
|------|------|-------------|-----------|-----|
| v0 baseline | bge-small 384 维（+结构） | 0.5575 | 0.4921（≈瞎猜） | 0.0764 |
| v1 结构+交互 | bge384 + 24 维中文结构/交互 | ≈0.56（无提升） | ≈0.49 | ≈0.06 |
| v2 历史成功率 TE | bge384 + eval_name 目标编码(α10) | 0.6412 | 0.5972 | **0.0539** |
| **v3 历史特征组合** | v2 + 跨模型难度 + 模型全局率(α20) | **0.6462 (+8.9pt)** | **0.6029** | 0.0586 |

**一句话**：真正拉动 AUC 的不是 prompt 文本特征，而是「**该模型在该任务类别上的历史成功率**」这一先验信号；overall AUC 在 ~0.65 处触及本数据集用"prompt + eval 级历史"能达到的天花板。

---

## v1（负结果，但有价值）：可解释结构 + 交互特征

- **构造了什么**：字符数、词数、平均词长、换行数、数字/问号个数、大写比例、是否含代码、选项数(ABCD)、few_shot 数、CJK 比例、是否中文、任务族 one-hot(6)，以及交互项（数字个数×数值推理、字符数×数值推理、选项数×选择题、代码信号×代码、CJK×中文文化、字符数×换行数）。
- **结果**：overall AUC 基本不动，结构/交互特征重要度接近 0。
- **为什么**：bge-small 的 384 维语义向量**已隐含**「这是数学题/选择题/中文/含代码」等浅层信息，手工再喂是**冗余**。
- **价值**：证明了「浅层 prompt 特征对已含语义的 embedding 无增量」，并给出可解释的中文特征重要度诊断。

## v2（正结果）：eval_name 目标编码（历史成功率）

- **构造了什么**：对每个 `(model, eval_name)`，用**训练集**里的成功率作为一维特征，喂给该 model 的 XGB。
  - 贝叶斯平滑：`rate = (成功数 + α·全局均值) / (样本数 + α)`，`α=10`；训练集未见的 eval_name 回退全局均值。
  - **model-specific**：每个模型各算一列（同一 prompt 对不同模型难度不同）。
- **不穿越的保证**：`rate` **只用 `train_proper` 统计**（不碰 cal / test）；expanding 折下 test 是"未来"，用"过去同类任务的历史成功率"预测未来，是标准的历史先验，非泄漏。生产环境同样可得（已知 query 属于哪个 benchmark 类别 + 历史数据）。
- **提升（同口径，校准后）**：
  - overall AUC **0.5606 → 0.6412（+8.1pt）**
  - 单模型 AUC 从 **~0.48（差于瞎猜）→ ~0.60**，解决了"单模型内部无区分力"的核心病灶
  - ECE **0.0635 → 0.0539**（同时改善，因为该特征本身就是概率刻度）
- **消融**：仅用 TE 这 1 维（不要 bge）raw AUC≈0.686、ECE≈0.073——单特征就逼近 384 维 bge，说明它是**主信号**。

## v3（微调）：历史特征组合 + 超参消融

在 v2 基础上系统消融（快速用 raw AUC 筛选，因 isotonic 单调校准不改变 AUC，再端到端确认）：

| 增量 | 效果 | 采用? |
|------|------|-------|
| **跨模型难度** cross（该 eval_name 所有模型平均成功率，train_proper 统计） | overall +0.003，单模型 +小幅 | ✅ |
| **模型全局率** mglobal（该 model 训练集平均成功率，常数列，帮跨模型排序） | 与 cross 协同 | ✅ |
| **α=10→20** | overall +0.006（样本少时更稳） | ✅ |
| 任务族 TE（6 类，粗粒度） | -0.005，与 eval_name 冗余 | ❌ 丢弃 |
| eval_name 样本数 log(count) | -0.003 | ❌ 丢弃 |
| XGB max_depth 5→3 | -0.005（欠拟合） | ❌ 保持 5 |
| holdout 0.3→0.15（训练数据更多） | raw AUC +0.01，但**校准后 overall 无增益且 ECE 0.054→0.074** | ❌ 保持 0.3 |

- **最终采用**：`te_eval + cross + mglobal`，α=20，holdout=0.3。
  端到端校准后 **overall AUC 0.6462 / ECE 0.0586 / 单模型均值 0.6029**——四个候选里 AUC 最高、ECE 几乎无损。
- **诚实结论**：v2→v3 只多挤出 +0.5pt overall。说明「prompt 文本 + eval 级历史」这条特征路线在本数据集已接近上限（~0.65）。要再明显突破，需要**query 级别**能区分"同一 eval_name 内谁对谁错"的信号（本数据里 prompt 文本几乎不携带），或换更强 embedding / 引入模型侧运行时信号。

## 附带发现

- 之前 AUC 只有 ~0.57 的一大原因是**小样本**：300 条时 baseline≈0.57，全量 1000 条 baseline 本身就更高。样本量是第一阶段的重要变量。
- **单模型 AUC 比 overall AUC 更贴近路由目标**：MILP 是"给定 query 在模型间做选择"，靠的是单模型内部的排序能力（0.49→0.60 的提升才是真正解决路由的关键），overall AUC 只是把所有 (query,model) 对混在一起的粗略参考。

## 开关

`config/config.yaml`：

```yaml
features:
  use_target_encoding: true    # 该 model×eval_name 历史成功率(主力特征)
  target_encoding_alpha: 20.0  # 平滑强度
  use_cross_difficulty: true   # 跨模型难度 + 模型全局率
predictor:
  calibration:
    holdout_frac: 0.3          # 校准集比例(实验证明 0.3 优于 0.15)
```

## 下一步可试方向（overall 已近上限, 重点转向 query 级信号）

1. **更强 embedding**：换 bge-large / e5，看语义是否能带来 query 级区分度。
2. **query 级难度**：prompt 长度分桶 × 任务族、是否多步推理等，与历史率交互。
3. **模型侧运行时信号**（若可得）：输出 token 数、自评置信度等。
4. 若 overall 确实到顶，则把优化目标转向**单模型 AUC** 与**第二阶段 Pareto 前沿**（校准质量 + 预算约束）。

## v4 扩展实验：数据量 / 超参 / 软标签 / KNN（2026-07-03）

> 口径说明：本节 raw AUC 实验使用 `initial_train=500, step=500`（每折训练窗口更大），与上文 v0–v3 的 `init=100, step=100` **不可直接横比**；但**同一协议内的相对排序**仍有效。

### 数据量 sweep（peek.csv 共 36,497 行，此前仅用 1,000）

| max_samples | overall AUC | 单模型均值 | ECE | 耗时 |
|-------------|-------------|-----------|-----|------|
| 1,000 | 0.7111 | 0.6777 | 0.1551 | 26s |
| 3,000 | 0.7047 | 0.6797 | 0.1598 | 73s |
| 6,000 | 0.7046 | 0.6770 | 0.1446 | 163s |
| 12,000 | 0.7027 | 0.6750 | 0.1193 | 570s |

**结论**：1k→12k **AUC 饱和甚至微降**，瓶颈是不可约噪声，不是数据量。加大 `max_samples` 不值得（编码 12k 要 ~10min，收益为零）。

### 超参 sweep @N=3000

| 配置 | overall | 单模型 | 采用? |
|------|---------|--------|-------|
| 基线 d5/n300/lr0.1 | 0.7047 | 0.6797 | — |
| d7/n500 | 0.7031 | 0.6771 | ❌ 过拟合倾向 |
| d9/n800/mcw5 | 0.7003 | 0.6752 | ❌ |
| **d6/n600/lr0.05/subs0.8/L2** | **0.7083** | **0.6834** | ✅ 轻正则最优 |

### 软标签 vs 二值 @N=3000

| 目标 | overall | 单模型 | 结论 |
|------|---------|--------|------|
| 二值 XGBClassifier | 0.7047 | 0.6797 | — |
| 软标签 reg:logistic | **0.6284** | **0.5932** | ❌ **大幅变差 −7.6pt** |

RouterBench 分数（0.1/0.25/0.75…）是部分匹配噪声，回归它反而伤害排序。**不采用。**

### KNN 成功率特征 @N=3000

| 配置 | overall | 单模型 | 结论 |
|------|---------|--------|------|
| 无 KNN | 0.7047 | 0.6797 | — |
| KNN k=20 | 0.7011 | 0.6761 | ❌ |
| KNN k=50 | 0.7047 | 0.6805 | ≈ 中性 |

与 bge+TE 冗余，**不采用。**

### v4 总论

在 v3 特征组合之上，**唯一值得落地的是轻正则超参**（depth6/n600/lr0.05/subs0.8/L2）；数据量、软标签、KNN 均无效或有害。

## AUC 0.71 算怎样？够不够高？

先把两个数字分清（很多人混在一起看）：

| 口径 | overall AUC | 单模型均值 | 含义 |
|------|-------------|-----------|------|
| **v3 正式口径**（init=100, step=100, **校准后**） | **0.646** | **0.603** | notebook / GitHub 应对齐这个 |
| **v4 raw 大窗口**（init=500, step=500, **未校准**） | **0.711** | **0.678** | 实验脚本里的 0.71 |

**0.71 不是「又涨了一截」**，主要是评估协议不同（训练窗口更大 + raw 未校准）。正式口径仍是 **~0.65**。

### 在 RouterBench / LLM routing 语境下够不够？

**结论：对「单实例成功率预测」来说中等偏上，对「路由决策」来说基本够用。**

1. **和 benchmark 对照**：RouterBench 论文里 KNN / MLP 等 baseline 的 per-model AUC 多在 **0.55–0.75** 区间；你单模型 ~0.60–0.68、overall ~0.65–0.71，属于**正常可用区间**，不是瞎猜（0.5），也远不到 oracle 级（≈1.0）。
2. **和路由目标的关系**：MILP 要的是**每条 query 上 4 个模型的相对排序**，不是每个概率都估准。你 notebook 里即使 AUC≈0.56，MILP 真实成功率仍达 **80.2%**（random 72%、always-cheap 64.7%）。预测一般 ≠ 路由无用。
3. **天花板原因**：同一 eval_name 内「这道题谁对谁错」有大量不可约噪声（aleatoric）；eval 级历史先验（TE）已吃掉主要可预测部分，再往上需要 query 级信号或更强模型（MF / 微调编码器），不是再堆数据能解决的。
4. **够不够放 GitHub？够。** 叙事应是：「阶段一 AUC 0.56→0.65，单模型从≈瞎猜到 0.60；阶段二 MILP 在预算下优于 random/cheap baseline，相对 oracle 的 gap 来自预测误差而非求解器。」

**一句话**：0.71（raw 大窗口）说明特征路线有效；正式 **0.65（校准后）** 在同类任务里**够用做路由 demo**，不必追求 0.9。后续重点是用 v3 预测**重跑 MILP Pareto**，用第二阶段 trade-off 补全展示。
